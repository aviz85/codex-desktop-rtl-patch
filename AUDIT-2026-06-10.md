# Deep Audit Report — codex-desktop-rtl-patch

- **Date:** 2026-06-10
- **Auditor:** Claude Code multi-agent workflow (architecture, security, quality, ops specialists + synthesis)
- **Scope:** Full repository at `/Users/aviz/codex-desktop-rtl-patch` (12 tracked files)
- **Verification:** The two most critical claims (dead-branch ternary in `src/codex-rtl-patch.js:65,93`; backup-once + unverified download fallback in `install.sh:106-135`) were re-verified against source before publication.

---

## Executive Summary

**Grade: B-**

This is a small, sharply-scoped Electron-app patching tool — one vanilla-JS injection payload plus bash (macOS) and PowerShell (Windows) installers — that solves the genuinely hard parts of in-place macOS patching (ElectronAsarIntegrity rehash, ad-hoc re-sign with library-validation disabled, bash-3.2 compatibility) correctly, with real dry-run modes, path-containment guards on every recursive delete, a substantive health check, clean git hygiene, and unusually honest documentation of its security trade-offs. The weaknesses cluster in three places: a **dead-branch ternary** (`stripped === "rtl" ? "rtl" : "rtl"`, duplicated in two functions) that silently disables the entire LTR-noise-stripping subsystem; a **backup-once restore path** that can install a stale `app.asar` from an older Codex version into an updated bundle, defeating the "fully reversible" claim; and a **supply-chain gap** where missing local files trigger a silent, integrity-unverified download of the patch JS from the *upstream* third-party repo rather than this fork. There are zero tests and no CI — exactly the gap that let the dead ternary ship. None of these undermine the core design, and all are cheap to fix.

---

## Tech Stack & Architecture Map

**Stack:** Vanilla browser-injection JavaScript (IIFE, ES5-style, ~270 lines) + bash 3.2-compatible macOS installers + PowerShell 5 Windows installers, patching an Electron `app.asar` via `@electron/asar@4.2.0` (fetched ad hoc — no `package.json`, no build, no tests, no CI). macOS side adds a launchd LaunchAgent watcher and `codesign`/`PlistBuddy` tooling.

**Layers:**

| Layer | Files | Role |
|---|---|---|
| Payload | `src/codex-rtl-patch.js` | DOM-attribute-only RTL direction shim injected as a `<script>` into Codex's `index.html`; MutationObserver + debounce drives re-processing |
| Installers | `install.sh`, `install.ps1` | macOS patches in place (asar repack → integrity rehash → ad-hoc re-sign); Windows copies Codex out of WindowsApps and patches the copy |
| Reversal | `uninstall.sh`, `uninstall.ps1` | Restore backups + re-sign (macOS cannot restore the notarized signature — ad-hoc only) |
| Health check | `check-macos.sh` | Verifies codesign, recomputes asar header sha256 vs `Info.plist` `ElectronAsarIntegrity`, checks patch presence — real checks, not theater |
| Autopatch | `autopatch/install-autopatch.sh`, `autopatch/README.md` | LaunchAgent (WatchPaths + RunAtLoad) that re-runs `install.sh --if-needed` after Codex updates; two documented modes (notify-only vs fully-automatic via FDA) |

Layering fits the project size well; no over-engineering. Idempotency guards, dry-run modes, and the launch-safety checker show real operational thinking. The structural anti-pattern is **triplication-and-drift**: the same concepts (entitlements plist, asar-header parsing, index.html injection, version constant, patched-detection predicate) are implemented 2–3 times across scripts with no shared source of truth.

## Findings by Severity

No Critical findings. The tool is local-only — no server, no API, no database, no secrets (confirmed by grep of working tree and git history).

### High

#### H1. Dead-branch ternary disables the entire LTR-noise-stripping subsystem
**File:** `/Users/aviz/codex-desktop-rtl-patch/src/codex-rtl-patch.js:65,93` *(verified)*
**Evidence:** Both `detectTextDir` and `detectElementDir` end with:
```js
var stripped = firstStrong(stripLeadingLTRNoise(text));
return stripped === "rtl" ? "rtl" : "rtl";
```
Both branches return `"rtl"`. The result of `stripLeadingLTRNoise()` (lines 50–57 — five regexes for domains, URLs, paths, backticked code, `$vars`) is computed and discarded. Consequence: any text containing even one Hebrew/Arabic character but starting with strong LTR (e.g., an English sentence quoting one Hebrew word) is forced `dir=rtl`. The identical mistake appears twice because the two functions are near-duplicates. The redundant `|| "rtl"` fallback at line 173 can also never trigger.
**Recommendation:** Decide the policy explicitly. Either use the stripped detection (`return stripped === "rtl" ? "rtl" : "ltr"`) or delete `stripLeadingLTRNoise` and the `stripped` computations entirely with a comment stating mixed text defaults to RTL. Then collapse the two near-duplicate functions into one taking a text-extractor, and pin the chosen behavior with a unit test. *(Flagged independently by all four auditors.)*

#### H2. Stale-backup restore: uninstall can install an old-version `app.asar` into a newer Codex bundle
**File:** `/Users/aviz/codex-desktop-rtl-patch/install.sh:130-135`, `uninstall.sh:55-57` *(verified)*
**Evidence:** Backups are created only `if [[ ! -f "$ASAR_BAK" ]]` ("backup already exists (kept)"). Flow: install (v1 backup taken) → Codex auto-updates to v2 → user re-runs `install.sh` per the README → backups stay at v1. A later `uninstall.sh` restores the v1 `app.asar` + `Info.plist` onto v2 native Electron frameworks/helpers — a version mismatch likely to crash or subtly break, defeating the README's "fully reversible" claim. Additionally, README line 120 and the `install.sh` header claim "timestamped backups"; the actual backups are fixed-name single copies (`app.asar.orig-backup`).
**Recommendation:** Refresh the backup whenever the current asar is unpatched: reuse the existing pristine-detection grep — `if [[ ! -f "$ASAR_BAK" ]] || ! grep -qa codex-rtl-patch.js "$ASAR"; then cp "$ASAR" "$ASAR_BAK"; fi` — and do the same for `Info.plist`. Store the Codex `CFBundleShortVersionString` alongside the backup and have `uninstall.sh` warn/abort on version mismatch (recommending app reinstall). Fix the "timestamped" claim in README and the script header.

#### H3. Zero automated tests and no CI for the core detection logic
**File:** `/Users/aviz/codex-desktop-rtl-patch/src/codex-rtl-patch.js` (repo-wide)
**Evidence:** No test file, no `package.json`, no `.github/workflows`, no shellcheck/PSScriptAnalyzer — 12 tracked files total. The payload contains pure, trivially unit-testable functions (`isRTLChar`, `hasRTL`, `firstStrong`, `stripLeadingLTRNoise`, `detectTextDir`, `detectElementDir`) with non-trivial bidi heuristics. A single assertion on LTR-leading mixed text would have caught H1. The only verification is the manual `check-macos.sh` and "run it and look."
**Recommendation:** Add a minimal `package.json` with a `node:test` suite covering Hebrew/Arabic/mixed/code-leading/URL-leading strings (export the pure functions via `if (typeof module !== "undefined") module.exports = {...}` before the document guard), plus a tiny GitHub Action running tests + shellcheck on `*.sh` + PSScriptAnalyzer on `*.ps1`.

### Medium

#### M1. Remote patch JS downloaded from the upstream third-party repo with no integrity check, then injected into Codex's webview
**File:** `/Users/aviz/codex-desktop-rtl-patch/install.sh:42,106-110`, `install.ps1:13,133-138` *(verified)*
**Evidence:** If `src/codex-rtl-patch.js` is missing locally, both installers silently `curl -fsSL` / `Invoke-WebRequest` it from `raw.githubusercontent.com/mnigli/codex-desktop-rtl-patch/main/...` — the **upstream fork parent, not this repo**, even though README line 131 tells macOS users to clone `aviz85/codex-desktop-rtl-patch`. No hash or signature check; the file is injected as a persistent `<script>` into Codex Desktop's `index.html`. Anyone who compromises (or merely changes) upstream `main` gets arbitrary JS in the user's Codex session — DOM, conversations, and whatever IPC the webview exposes. The README's primary Windows path is also `irm .../install.ps1 | iex` from the same unpinned branch. This also breaks reproducibility: the fork's local payload (v0.1.0) can drift from upstream `main` at any time.
**Recommendation:** Best: remove the network fallback and hard-fail when the local file is missing — the documented install path is a clone, so a missing file means a broken checkout. Otherwise: point `PATCH_JS_URL` at the fork users were told to clone, pin to a commit SHA (not `main`), and verify a known sha256 before injecting (`echo "<sha256>  $PATCH_JS" | shasum -a 256 -c`).

#### M2. MutationObserver reprocesses the entire document every 60ms during streaming
**File:** `/Users/aviz/codex-desktop-rtl-patch/src/codex-rtl-patch.js:238-252`
**Evidence:** The observer ignores mutation targets entirely and calls `processAll(document.body)` after a 60ms debounce. `processAll` runs 6+ full-document `querySelectorAll` passes; `processLeafContainers` (line 163) iterates every `div, span, button, a, label`, calling a 14-selector `hasBlockChild` query plus `textContent` on each; `applyDir` unconditionally rewrites `dir`/`style` attributes even when unchanged, causing style-recalc churn. In a chat app streaming tokens (constant characterData mutations), this is an O(whole-DOM) read+write cycle ~16×/second on long transcripts — the design's main scalability bottleneck. The 60ms debounce and `text.length < 2` threshold (line 172) are unexplained magic numbers.
**Recommendation:** Collect `mutation.target` / `addedNodes` roots into a Set during the debounce window, dedupe to top-most ancestors, and process those subtrees instead of `document.body`; keep a low-frequency full pass as fallback. In `applyDir`, skip writes when `el.getAttribute("dir") === dir`. Name the constants (`DEBOUNCE_MS`, `MIN_TEXT_LEN`).

#### M3. Documented Mode B grants Full Disk Access to `/bin/bash`, weakening TCC system-wide
**File:** `/Users/aviz/codex-desktop-rtl-patch/autopatch/install-autopatch.sh:67-80`, `autopatch/README.md:26-33`
**Evidence:** For fully-automatic re-patching, the agent installer prints instructions and **auto-opens the Privacy pane** to add `/bin/bash` to Full Disk Access. Once granted, any script executed via `/bin/bash` by any process inherits FDA — malware no longer needs its own TCC grant. The README honestly warns the grant is broad, but the script actively nudges users toward the unsafe option. Compounding: the LaunchAgent's `ProgramArguments` points at `install.sh` inside the user-writable repo clone, so with FDA granted, anything that can write that file runs with Full Disk Access at next Codex update/login.
**Recommendation:** Replace the `/bin/bash` grant with a dedicated wrapper at a fixed, admin-owned path (or a tiny helper binary) and grant FDA to that executable only. At minimum, have the agent verify a checksum of `install.sh` before executing it, and stop auto-opening the FDA pane so Mode A (notify-only) stays the path of least resistance.

#### M4. Ad-hoc re-sign strips library validation from an app holding auth credentials; uninstall leaves it that way
**File:** `/Users/aviz/codex-desktop-rtl-patch/install.sh:226-253`, `uninstall.sh:65-86`
**Evidence:** The patched bundle is re-signed ad-hoc with `disable-library-validation`, `allow-jit`, and `allow-unsigned-executable-memory` — required for this approach, but the patched Codex.app will then load any dylib (no team-ID check), and Codex stores auth in `~/.codex`, so local malware can inject code into a process holding the user's OpenAI session. Notably, `uninstall.sh` restores the original asar but re-signs **ad-hoc with the same weakened entitlements** (it cannot restore the notarized OpenAI signature), mentioning this only in passing.
**Recommendation:** Inherent trade-off, but surface it at install time with an explicit interactive confirmation ("this disables code-signing protections on Codex — continue? [y/N]", flag-skippable). Have `uninstall.sh` print a prominent recommendation to reinstall Codex to restore the notarized signature.

#### M5. Entitlements plist and `find_codex()` duplicated verbatim between install.sh and uninstall.sh
**File:** `/Users/aviz/codex-desktop-rtl-patch/install.sh:231-245,58-68`, `uninstall.sh:66-80,26-34`
**Evidence:** The 7-key codesign entitlements heredoc and the Codex locator (CODEX_APP override → /Applications → ~/Applications → mdfind) are copy-pasted in both scripts. If a future Codex build needs an extra entitlement added only to `install.sh`, `uninstall.sh` re-signs the restored bundle with the wrong set and the "restored" app may not launch — exactly the failure mode the uninstall comments warn about, and hard to debug.
**Recommendation:** Extract to shared artifacts (a checked-in `src/codex-resign.entitlements` file + a `scripts/common.sh` with `find_codex`/`resign`) sourced by both, falling back to the heredoc only in the curl-pipe scenario.

#### M6. No rollback on mid-install failure: failed codesign leaves a patched-but-unlaunchable app
**File:** `/Users/aviz/codex-desktop-rtl-patch/install.sh:190,208,246-247`
**Evidence:** The patched asar is installed (line 190) and `Info.plist` edited (line 208) **before** re-signing (line 246). If `codesign` fails, the script dies ("bundle is patched but unsigned and will not launch") leaving Codex broken — the EXIT trap (line 139) only cleans the temp dir; it does not restore the backups it just made. Same exposure for a `PlistBuddy` failure between steps.
**Recommendation:** Set a flag once `$ASAR` is overwritten and extend the trap: on non-zero exit with the flag set, `cp "$ASAR_BAK" "$ASAR" && cp "$PLIST_BAK" "$PLIST"`, then attempt the ad-hoc re-sign.

#### M7. Two divergent implementations of asar header parsing / three patched-detection predicates
**File:** `/Users/aviz/codex-desktop-rtl-patch/check-macos.sh:8-9`, `install.sh:88,202-206`
**Evidence:** `install.sh` computes the integrity hash via `@electron/asar.getRawHeader().headerString`; `check-macos.sh` hand-rolls the asar header format with raw fs reads (16-byte pickle header, `readUInt32LE(12)`) in two near-identical inline node one-liners. Patch detection also differs: `install.sh --if-needed` greps the whole asar binary; `check-macos.sh` greps only the header string. If the asar format or library semantics change, the installer and health check can disagree — the checker reports MISMATCH on a healthy app or vice versa.
**Recommendation:** Factor the header-read into one small checked-in `.js` helper used by both scripts; standardize on one patched-detection predicate.

### Low

#### L1. Version constants drifted across three places (0.2.0 / 0.1.0 / 0.1.0)
**File:** `install.sh:23` (`PATCH_VERSION="0.2.0"`), `install.ps1:19` (`'0.1.0'`), `src/codex-rtl-patch.js:8` (`"0.1.0"`)
**Evidence:** No single source of truth; "RTL patch v0.2.0 installed" on macOS reports a version the actual JS payload doesn't carry. Worse, the JS version is the runtime idempotency/re-injection guard (line 7), so a genuinely updated payload with an unchanged version string would silently refuse to apply.
**Recommendation:** Keep the version only in `src/codex-rtl-patch.js` and have installers read it (grep `__codexRtlPatchVersion`) at run time; bump it on every payload change.

#### L2. Autopatch success message claims a "6h fallback" that no longer exists
**File:** `/Users/aviz/codex-desktop-rtl-patch/autopatch/install-autopatch.sh:64`
**Evidence:** Line 64 prints "Re-applies the RTL patch after every Codex update (+ 6h fallback)", but the generated plist (lines 34–57) has only `WatchPaths` + `RunAtLoad` — no `StartInterval`. Commit `e36b919` ("drop 6h interval (WatchPaths-only)") removed it but missed this echo. The script/README also reference a "Windows scheduled-task watcher" analog that doesn't exist in this repo.
**Recommendation:** Delete "(+ 6h fallback)" and drop or correctly attribute the Windows-watcher cross-reference.

#### L3. Autopatch agent can race a live Codex auto-update
**File:** `/Users/aviz/codex-desktop-rtl-patch/autopatch/install-autopatch.sh:50-51`
**Evidence:** The LaunchAgent fires the instant `app.asar` changes and runs `install.sh`, which immediately force-quits Codex (osascript quit, then `pkill` after 10s) and rewrites asar + plist + re-signs. During an auto-update the updater may still be writing other bundle files, so the agent can patch a half-replaced bundle or kill the updater mid-flight. No settle delay or update-in-progress detection. (Credit: `autopatch/README.md` honestly marks this mode "designed but not yet end-to-end verified".)
**Recommendation:** Add a settle guard to the `--if-needed` path: wait until `app.asar` mtime has been stable ~30–60s and skip if a Codex updater process is running.

#### L4. `processInputs` forces `dir="ltr"` on empty/undetected inputs
**File:** `/Users/aviz/codex-desktop-rtl-patch/src/codex-rtl-patch.js:178-194`
**Evidence:** `detectTextDir` returns `null` for empty text, but the else branch treats `null` as LTR and hard-writes `dir="ltr"` + `direction:ltr` + `text-align:left` — stamping an empty composer LTR in an RTL UI and clobbering any host-app dir. `applyDir` already has a correct null → remove-attributes path (lines 113–118); the inputs path doesn't reuse it.
**Recommendation:** Mirror `applyDir`'s null branch (remove the patch's attributes) and reuse `applyDir` for inputs to keep one write path.

#### L5. `check-macos.sh` falsely reports MISMATCH when node is missing; deprecated `launchctl load/unload`
**File:** `/Users/aviz/codex-desktop-rtl-patch/check-macos.sh:15`, `autopatch/install-autopatch.sh:59-60`
**Evidence:** The hash one-liner ends with `|| echo ""` and the script never checks for node (unlike `install.sh:70`); without node, AH is empty while PH is set and line 18 prints "✗ MISMATCH — Codex will SIGTRAP at startup" — a false alarm from a health check whose job is accuracy. Separately, the autopatch installer uses `launchctl unload/load`, deprecated since macOS 10.10.
**Recommendation:** Fail fast with a clear message if `command -v node` is absent (or distinguish "could not compute" from "mismatch"). Switch to `launchctl bootout gui/$UID` / `bootstrap gui/$UID`.

#### L6. Duplicated index.html injection logic in two languages with divergent output
**File:** `install.sh:165-181` (inline python3, `\n`), `install.ps1:141-173` (`\r\n`, different regex escaping)
**Evidence:** The insert-script-tag-before-main-bundle logic exists twice; when Codex changes its `index.html` shape (the project's stated #1 breakage risk), both must change in lockstep, and today they already produce different file contents (CRLF vs LF) for the same input.
**Recommendation:** Move injection into one small Node script invoked by both installers (Node is already a hard requirement on both platforms) — also removes the python3 dependency on macOS.

#### L7. `npm install` of `@electron/asar` runs lifecycle scripts with unpinned transitive deps
**File:** `install.sh:24,147`, `install.ps1:20,193,205`
**Evidence:** Top-level `@electron/asar@4.2.0` is pinned (good), but there's no lockfile and no `--ignore-scripts`, so transitive deps resolve at install time and any postinstall script executes — including during unattended LaunchAgent runs in Mode B.
**Recommendation:** Add `--ignore-scripts` (asar needs no native build) and consider a committed `package-lock.json` + `npm ci`.

#### L8. `pgrep`/`pkill -f` with unescaped path treated as regex
**File:** `install.sh:113-124`, `uninstall.sh:46-53`
**Evidence:** `$APP` (from `CODEX_APP` env or `mdfind`) is passed to `-f` as a regex; metacharacters could match and kill unrelated processes of the invoking user. Robustness more than security (no privilege boundary crossed).
**Recommendation:** Escape the path or match by bundle ID (`osascript -e 'quit app id "com.openai.codex"'`).

#### L9. Transient write-test file created inside the signed bundle
**File:** `install.sh:96-102` *(verified — write-test cleanup at line 102)*
**Evidence:** Writability is probed by creating `$RES/.rtl-write-test` inside `Codex.app/Contents/Resources` — while the script's own comment (lines 78–79) warns that a stray file inside `Contents/` invalidates the code signature. A Ctrl-C between creation and `rm -f` leaves the stray file breaking the still-unpatched app's signature.
**Recommendation:** Guarantee removal via the EXIT trap (or probe without touching the bundle interior).

### Positive findings (keep these patterns)

- **Path-containment guards on all recursive deletes:** every `Remove-Item -Recurse` is preceded by `Assert-UnderPath` (canonicalize + verify under `%LOCALAPPDATA%\OpenAI` or temp root); the macOS trap removes only its own `mktemp` dir. (`uninstall.ps1:20-30`, `install.ps1:46-53,90,181-182`)
- **Honest privilege-escalation documentation:** `autopatch/README.md` explicitly presents the two-mode design, flags the FDA grant as broad, and marks Mode B "designed but not yet end-to-end verified". Rare candor.
- **Substantive health check:** `check-macos.sh` verifies codesign, recomputes the real asar header sha256 against `Info.plist`, and checks patch presence — real checks, not theater.
- **Clean repo hygiene:** 12 tracked files, no binaries, no secrets in tree or git history, sane `.gitignore`, pinned asar version, forced TLS 1.2 on Windows, genuine dry-run modes gating every mutation.
